# 开发方案归档（`doc/dev/plans`）

[中文](./README.md)

本目录用于存放**与代码提交绑定的方案文档**，尤其适用于 AI 辅助开发流程。规范全文见 [`doc/development.md`](../../development.md) 中的「AI 生成代码贡献规范」。

## 何时需要建方案目录

- 新功能、重构、跨模块改动、行为或 API 契约变更
- 使用 AI 生成或大幅改写代码的提交
- 维护者要求留档的修复或实验

单文件 typo、纯注释、无行为变化的格式化等，可由维护者豁免。

## 子目录命名

```text
{Unix毫秒时间戳}_{简要描述}_{32位十六进制UUID}
```

| 段 | 说明 |
|----|------|
| 时间戳 | Unix **timestamp**（毫秒），如 `1782301810000` → `2026-06-24 19:50:10`（UTC+8） |
| 简要描述 | 英文连字符或简短中文，概括本次变更 |
| UUID | 128 位 UUID 的 32 位十六进制（无 `-`），防重名 |

示例目录名：

```text
1782301810000_examples-html-app-entry_f8e3b2a14c5d6789e0f1a2b3c4d5e6f7
```

## 目录内文档

每个子目录应覆盖下列三类内容（**至少两个 `.md` 文件**，可按模板拆分或合并）：

| 类型 | 建议文件名 | 内容 |
|------|------------|------|
| 提示词与上下文 | `01-prompt-and-context.md` | 核心提示词摘要；人类↔AI 会话摘要（目标、约束、决策、弃案） |
| 需求评估 | `02-requirements-assessment.md` | 背景、范围、非目标、风险、验收标准 |
| 方案设计 | `03-solution-design.md` | 架构、接口、影响面、测试与回滚 |

若仅使用两个文件，可将「需求评估」与「方案设计」合并为 `02-requirements-and-design.md`（见下方示例）。

## 测试

变更涉及 `core/`、`domains/` 等可测逻辑时，应在 [`tests/`](../../../tests/) 下补充 `*.test.mjs`，并在方案中写明覆盖点；纯 HTML 示例等可仅列手动验收项。

## 人类评审

方案合并前须在目录内或 PR 中记录：

- 评审人
- 评审日期
- 结论（通过 / 修改后通过 / 驳回）
- 必要时的修改说明
